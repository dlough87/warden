{# ── 1. Overview ── #}

1. Overview

Warden is an automated media library hygiene tool. It connects to your Radarr and Sonarr instances to discover every movie and TV show in your collection, queries Tautulli (and Plex as a fallback) for watch history, evaluates each item against a set of configurable rules, and automatically removes items that no longer belong in your library — while giving you full control over the process.

Key principles:

  • Non-destructive by default — Warden ships with Dry Run mode enabled. Nothing is deleted until you turn it off.
  • Grace period built in — Condemned items sit on Death Row for a configurable number of days before deletion, giving you time to review and pardon anything that shouldn't be removed.
  • Fully reversible until deletion — Pardoning an item permanently protects it from all future rule evaluations.
  • Exclusion list integration — Deleted items are added to Radarr/Sonarr's import exclusion list so they aren't auto-imported again.
{# ── 2. How It Works ── #}

2. How It Works

Each scan runs the following pipeline:

1. Fetch media — Radarr provides all movies; Sonarr provides all TV series
↓
2. Build watch maps — Tautulli (primary, threshold-filtered) and Plex (fallback) provide per-item watch history
↓
3. Evaluate rules — Each item is tested against every enabled rule; a match on any rule condemns the item
↓
4. Update database — All status changes are written in a single batch transaction
↓
5. Send notifications — Newly condemned items, reminders, and deletions are dispatched to configured agents
↓
6. Execute deletions — Items whose Death Row sentence has expired are deleted from Radarr/Sonarr (skipped in Dry Run)

Item Status Lifecycle

StatusMeaningTransitions to
OKNo rules matched. Item is safe.Condemned (if rules match on next scan)
CondemnedMatched one or more rules. Death row timer is running.OK (if rules no longer match), Pardoned (manual), Deleted (timer expires)
PardonedManually exempted. Never re-evaluated by rules.No automatic transitions
DeletedFiles removed from disk; added to import exclusion list.No transitions (terminal state)

Important: If a condemned item no longer matches any rule (e.g. the rule is disabled, or watch data updated), it is automatically cleared back to OK on the next scan. Its death row timer resets if it is condemned again.

{# ── 3. Getting Started ── #}

3. Getting Started

Follow these steps to get Warden running against your library.

Step 1 — Connect your services

Navigate to Settings → Connections and fill in the details for each service.

ServiceWhat you needWhere to find it
RadarrInternal URL + API keyRadarr → Settings → General → Security → API Key
SonarrInternal URL + API keySonarr → Settings → General → Security → API Key
PlexInternal URL + X-Plex-TokenSee Plex support article
TautulliInternal URL + API keyTautulli → Settings → Web Interface → API Key

Use the Test button next to each service to verify the connection before saving.

Docker tip: If Warden and your services run on the same Docker network, use their service names as hostnames (e.g. http://radarr:7878, http://sonarr:8989). Set Public URLs to your externally accessible addresses — these are used only for the "Open In" links in the UI.

Step 2 — Review the default rules

Warden ships with three starter rules. Go to Rules and review them before your first live scan.

RuleWhat it targets
Never WatchedItems added more than 18 months ago that have never been played
Old & Low RatedItems in library 5+ years, not watched in 18+ months, rated below 7.0
Very OldItems in library 8+ years, not watched in 5+ years, rated below 8.0

Disable any rule you don't want to apply and adjust the thresholds to suit your library. See Section 6 for a full explanation of all rule criteria.

Step 3 — Set up a notification agent

Go to Settings → Notification Agents → + Add Agent and configure at least one agent so you receive alerts before items are deleted. Discord webhooks are the most popular choice for home setups. See Section 10 for full details.

Step 4 — Run a test scan in Dry Run mode

Click Run Scan on the Dashboard. Because Dry Run is enabled by default, nothing will be deleted. After the scan completes, check Death Row and Candidates to see which items would be condemned. Pardon anything that looks wrong.

Step 5 — Disable Dry Run when you're ready

Once you're confident in your rules, go to Settings and uncheck Dry Run mode. The orange DRY RUN badge in the navigation bar will disappear. From this point, items that complete their death row sentence will be permanently deleted.

Warning: Deletion is permanent. Files are removed from disk and the item is added to the Radarr/Sonarr import exclusion list. Always review Death Row regularly and pardon items you want to keep.

{# ── 4. Account Security ── #}

4. Account Security

All account security settings are managed from your Profile page, accessible by clicking your username in the top-right navigation menu.

Username & Password

The Account Details section lets you change your display name. The Change Password section requires your current password before a new one is accepted. Passwords must be at least 8 characters.

Two-Factor Authentication (TOTP)

Warden supports time-based one-time passwords (TOTP) compatible with any standard authenticator app — Google Authenticator, Authy, 1Password, and others.

To enable 2FA:

  1. Go to Profile → Two-Factor Authentication → Set Up Two-Factor Auth
  2. Scan the QR code with your authenticator app
  3. Enter the 6-digit code to confirm it's working
  4. Save your backup codes — each can be used once if you lose access to your authenticator

Once enabled, every login requires your password and a 6-digit code. To disable 2FA, you must confirm with both your password and a valid code.

Keep your backup codes safe. If you lose your authenticator and have no backup codes, you will be locked out. Store them in a password manager.

Passkeys

Passkeys are a modern, phishing-resistant alternative to username + password + 2FA. A single touch with Face ID, Touch ID, Windows Hello, or a hardware security key replaces the entire login flow.

Prerequisites

  • Warden must be accessed over HTTPS — browsers will not allow passkeys over plain HTTP
  • Passkey Domain must be set in Settings → General to the exact hostname you use (e.g. warden.example.com) — no https://, no port number
  • Self-signed certificates will not work — you need a certificate from a recognised CA (Let's Encrypt is free and works well)

Getting HTTPS

If you're not already behind a reverse proxy or tunnel, there are two straightforward options:

Option A — Cloudflare Tunnel (no ports to open)
  1. Create a free Cloudflare account and add your domain
  2. Install cloudflared on your host or as a Docker container
  3. Create a tunnel in the Cloudflare Zero Trust dashboard and point it at http://warden:8787 (or http://localhost:8787 if not using Docker networking)
  4. Cloudflare issues a trusted certificate automatically — no cert management needed
  5. Set Passkey Domain to the public hostname Cloudflare assigns (e.g. warden.example.com)
Option B — Caddy reverse proxy (auto Let's Encrypt)

Caddy provisions and renews Let's Encrypt certificates automatically. Add it alongside Warden in your docker-compose.yml:

services:
  caddy:
    image: caddy:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data
    restart: unless-stopped

volumes:
  caddy_data:

Create a Caddyfile next to your docker-compose.yml:

warden.example.com {
    reverse_proxy warden:8787
}

Replace warden.example.com with your actual domain. Ports 80 and 443 must be reachable from the internet for Let's Encrypt to issue a certificate. Set Passkey Domain to the same hostname.

Both options require a real domain name pointed at your server. Passkeys cannot be used on bare IP addresses or localhost (except localhost itself, which browsers treat as a special case for development).

Registering a passkey

  1. Set your Passkey Domain in Settings → General and save
  2. Go to Profile → Passkeys
  3. Give the passkey a recognisable name (e.g. MacBook Touch ID, YubiKey)
  4. Click Add a Passkey and follow your browser/device prompt

You can register as many passkeys as you like — one per device is recommended so you can revoke them individually.

Signing in with a passkey

On the login page, click Sign in with a Passkey. Your browser will prompt you to authenticate with the device's biometrics or security key. No username or password is needed.

Removing a passkey

Go to Profile → Passkeys, find the passkey you want to remove, and click Remove. The passkey is immediately invalidated. Removing your only passkey does not affect password-based login.

Passkey Domain must not change. If you change the Passkey Domain setting after registering passkeys, all existing passkeys will stop working. Remove them all from Profile first, then update the domain and re-register.

{# ── 6. Rules ── #}

6. Rules

Rules define which items become candidates for deletion. An item must satisfy every criterion within a single rule to be condemned — all conditions within a rule are AND logic. A match on any rule is sufficient to condemn an item.

Criteria reference

CriterionTypeDescription
Added monthsIntegerItem must have been in the library for at least this many months. Warden uses the earliest known date across Radarr/Sonarr and Plex.
Unwatched monthsIntegerLast watched date must be more than this many months ago — or the item must have never been watched at all.
Max ratingDecimalIMDb (movies) or TVDB (shows) rating must be at or below this value. Leave blank to ignore rating.
Min ratingDecimalRating must be at or above this value. Useful to protect high-rated content from other criteria (combine with other rules).
Max playsIntegerTotal play count must be below this number. E.g. 1 targets items watched exactly once long ago.
Min size (GB)DecimalFile must be at least this size. Target only large files occupying disproportionate disk space.
Exclude genresComma-separatedItems with any of these genres are skipped by this rule. E.g. Animation, Documentary.
Include genresComma-separatedOnly items containing at least one of these genres are evaluated. All other genres are ignored by this rule.
Media typeMovie / Show / BothRestrict this rule to movies, TV shows, or both.

Rule priority & ordering

Rules are evaluated in Sort Order sequence (set when editing a rule). The order only affects which rule name appears first in the "Rules Matched" column — all matching rules are recorded. Edit the Sort Order field to control which rule is displayed as the primary match.

Example rules

GoalSettings
Remove movies never watched after 2 yearsAdded months: 24, Unwatched months: 24, Media type: Movie
Remove large files not touched in 3 yearsAdded months: 36, Unwatched months: 36, Min size: 20
Protect all Animation regardless of other rulesCreate a rule with Include genres: Animation and Max rating: 10 — this alone won't condemn anything, but other rules should Exclude genres: Animation
Clear out low-rated old contentAdded months: 48, Unwatched months: 12, Max rating: 6.5
{# ── 7. Death Row ── #}

7. Death Row

Death Row shows every item that has been condemned and is awaiting deletion. The countdown timer is based on the Death Row Days setting (default 30 days). An item's timer starts the day it is first condemned and does not reset if it continues to match rules on subsequent scans.

Days Left column

Items with 7 or fewer days left are highlighted in red. Items due for deletion today show "Today" — they will be deleted on the next scan that runs in live mode.

Sorting

Click any column header to sort by that field. Click again to reverse the direction. The default sort is by Days Left ascending (soonest deletion first).

Pardoning an item

To permanently protect an item, enter an optional reason in the text box next to it and click Pardon. Pardoned items:

  • Are moved to Pardoned status immediately
  • Are never re-evaluated by any rule on future scans
  • Remain visible in the Library with a green "Pardoned" badge
  • Can still be viewed in detail, including all watch statistics

Tip: There is no "unpardon" button in the current UI. If you pardon an item by mistake, it will simply be protected forever — which in practice is harmless.

{# ── 8. Candidates ── #}

8. Candidates

The Candidates page shows the same condemned items as Death Row, but grouped by the rule that matched them. This is useful for auditing your rules: you can quickly see which rule is responsible for the most deletions and whether the results look correct.

Each rule group has its own sortable table. Use the column headers to sort within each group. The sort applies to all groups simultaneously.

Candidates vs Death Row: These pages show identical items. Candidates groups by rule for rule auditing; Death Row groups all items together with the countdown timer for deletion scheduling.

9. Library

The Library is a browseable, searchable, sortable view of your entire media collection as known to Warden.

Filters

StatusAll active (default, hides deleted) / OK / Condemned / Pardoned / Deleted

Sorting

Click any column header (Title, Year, Rating, Size, Added, Last Watched, Plays, Status) to sort by that field. Click again to reverse. Sort is preserved in the URL so you can bookmark views.

Item detail page

Click any item's title to open its detail page. This shows:

  • Watch stats — total plays, last watched date, maximum watch percentage, size on disk
  • Library info — added date, runtime, certification, studio/network, series status
  • File info — quality profile, size, file path
  • Warden status — matched rules, death row date, days remaining, pardon form (for condemned items)
  • Open In links — quick links to IMDb, TMDB/TVDB, Radarr/Sonarr, and Plex (hidden for deleted items)
{# ── 10. Notifications ── #}

10. Notification Agents

Warden supports multiple independent notification agents. Each agent has its own credentials, enabled/disabled state, event subscriptions, and (for reminders) day thresholds. You can have as many agents as you like — for example, a Discord agent for condemned items going to one channel and a Pushover agent for deletions going to your phone.

Managing agents

Go to Settings → Notification Agents. From there you can:

  • Click + Add Agent to create a new agent (you'll be prompted to choose the type first)
  • Click Test to send a test message to an existing agent
  • Click Edit to modify an agent's configuration
  • Click Enable / Disable to temporarily silence an agent without deleting it
  • Click Delete to permanently remove an agent

Agent types

Discord

Sends rich embed messages to a Discord channel via a webhook. Messages are colour-coded by event type and include item lists grouped by rule.

FieldDescription
Webhook URLThe full Discord webhook URL. Create one in Discord: Channel Settings → Integrations → Webhooks → New Webhook.
Mention (optional)A Discord user or role mention prepended to condemned and deletion messages. Format: <@USER_ID> or <@&ROLE_ID>.
Bot Avatar URL (optional)URL of an image to use as the webhook's avatar, overriding the default.

Pushover

Sends push notifications to your Pushover-connected devices.

FieldDescription
User KeyYour Pushover user or group key, found at pushover.net
API TokenThe API token for your Pushover application (create one at pushover.net/apps/build)
PriorityLowest / Low / Normal / High — controls notification urgency on your device
Device (optional)Specific device name to target. Leave blank to notify all your devices.

ntfy

Sends notifications via the ntfy push notification service. Works with the public ntfy.sh server or your own self-hosted instance.

FieldDescription
Server URLBase URL of your ntfy server. Use https://ntfy.sh for the public service.
TopicThe ntfy topic name to publish to (acts like a channel). Choose something unique.
Access Token (optional)Required only if your topic is access-controlled. Generate in your ntfy account settings.
PriorityMin / Low / Default / High / Urgent

Apprise

Routes notifications through a self-hosted Apprise API server, which supports 100+ notification services.

FieldDescription
Server URLBase URL of your Apprise API server (e.g. http://apprise:8000)
Tags (optional)Comma-separated Apprise tags to target. Leave blank to notify all configured URLs in Apprise.

JSON Webhook

Posts a raw JSON payload to any HTTP endpoint — useful for home automation, custom scripts, or services not covered by the other agent types.

FieldDescription
URLThe endpoint to POST (or GET) to
HTTP MethodPOST (default) or GET
Additional Headers (optional)JSON object of extra headers. Example: {"Authorization": "Bearer token123"}

Events

EventWhen it fires
🚨 New items condemnedAn item is placed on death row for the first time in this scan (not on re-scans where it was already condemned)
⏳ Death row remindersA condemned item hits one of the configured day thresholds before its deletion date. Each threshold fires at most once per item per death row sentence.
🗑️ Items deletedOne or more items are deleted (or would be deleted in Dry Run). Includes total space freed.
✅ Clean scanA scan completes with zero newly condemned items

Death row reminders

When you enable the Death row reminders event on an agent, a text field appears below the checkbox labelled "At … days remaining". Enter a comma-separated list of day thresholds, e.g. 7, 3, 1. Warden will send a reminder notification each time a condemned item crosses one of these thresholds.

Each threshold fires once per item per condemnation. If an item is cleared (rules no longer match) and then condemned again on a future scan, the reminder thresholds reset and will fire again.

Example: Set thresholds to 7, 1. An item condemned today with a 30-day death row period will trigger a "7 days left" reminder on day 23 and a "1 day left" reminder on day 29.

{# ── 12. Support Tools ── #}

12. Support Tools

The Support section (linked in the top navigation) contains four tools for troubleshooting and maintenance.

Documentation

This guide.

Audit Log

A chronological record of significant actions taken in Warden — settings changes, scans triggered, items pardoned, passkeys registered, and so on. Each entry shows a timestamp, action description, and the IP address it originated from. The audit log is useful for understanding what changed and when, especially in a shared environment.

The log retains the most recent 200 entries by default and is displayed newest-first.

Server Log

The last 500 lines of Warden's application log output, auto-refreshing every 10 seconds. This is the first place to look when diagnosing connection errors, scan failures, or notification delivery issues. Errors appear in red; warnings in yellow.

You can also access the full log stream outside the UI with docker logs -f <container_name>.

Backup & Restore

Two backup types are available:

TypeWhat's includedUse for
Config backup (.json)Settings, rules, and notification agentsRoutine backups, migrating configuration to a new instance
Full backup (.db)Everything — config, media library, scan history, audit logBefore major upgrades, disaster recovery

Restoring a config backup

Upload a .json config backup using the Restore form on the Backup page. This replaces your current settings, rules, and notification agents. Account credentials (username, password, 2FA, passkeys) are never overwritten by a config restore.

Restoring a full backup

Full database restores must be done at the container level — the UI cannot replace the live database file:

  1. Stop the Warden container
  2. Replace /data/warden.db with your backup file
  3. Start the container again
{# ── 13. Troubleshooting ── #}

13. Troubleshooting

Items aren't being condemned

  • Check that at least one rule is enabled (Rules page — disabled rules are greyed out)
  • Verify your criteria are realistic for your library. Open the Library page, find an item you expect to be condemned, and check its Added date, Last Watched, and Rating against your rule thresholds.
  • Make sure a scan has completed after you saved your rule changes. The scan runs on a schedule; use Run Scan on the Dashboard to trigger one immediately.

Watch history shows "Never" for items I've watched

  • Check the Tautulli connection (Settings → Connections → Test). If Tautulli is unreachable, Warden falls back to Plex, which may not have history for all items.
  • Tautulli only records sessions from the date it was installed. Items watched before Tautulli was set up will show "Never" in Tautulli — Plex's lastViewedAt is used as a fallback in this case.
  • The watch threshold may be set too high. If you stopped a video at 79% and your threshold is 80%, it won't count. Lower the threshold in Settings.

Notifications aren't being sent

  • Use the Test button on the agent in Settings → Notification Agents. This sends a test message immediately and shows any error inline.
  • Ensure the agent is enabled (the Enable/Disable button on the agent card).
  • Make sure the relevant event is checked in the agent's event list.
  • Check the Server Log (Support → Server Log) for error messages from the notification dispatcher.
  • For Discord: verify the webhook URL is valid and the webhook hasn't been deleted in Discord.
  • For ntfy: if using a protected topic, ensure the access token is correct.

An item keeps getting condemned after I pardon it

This should not happen — pardoned items are permanently exempt from all rule evaluation. If you believe this is occurring, check the Audit Log (Support → Audit Log) for any "Item pardoned" entries and verify the pardon was saved successfully.

Scan completes instantly with no results

  • Check that Radarr and Sonarr are connected and returning data (Settings → Connections → Test).
  • Open the Server Log — it will show how many items were fetched from Radarr/Sonarr and how many rules were evaluated.

Added dates look wrong (items appear newer than they are)

Warden takes the minimum date across Radarr/Sonarr, Plex library added date, and Plex last-viewed date. This guards against library resets pushing dates forward. If an item still shows an incorrect date, it may mean neither Radarr/Sonarr nor Plex has an accurate history for it.

Deleted items still appear in Radarr/Sonarr

Warden deletes the item and adds it to the import exclusion list via the Radarr/Sonarr API. If the item reappears, check whether something else (e.g. an import list or auto-search) is re-adding it. The exclusion list should prevent this, but if the TMDB/TVDB ID differs from what Warden recorded, the exclusion may not match.

Passkey registration or sign-in fails

  • Confirm Passkey Domain in Settings → General is set to your exact hostname (e.g. warden.example.com) — no https:// prefix, no port number.
  • Warden must be accessed over HTTPS with a trusted certificate. Cloudflare Tunnel and Caddy both satisfy this. Self-signed certs will not work.
  • The domain in the browser's address bar must exactly match the Passkey Domain setting. Passkeys registered on one URL will not work on another (e.g. registering on warden.example.com then trying to sign in via an IP address will fail).
  • If you changed the Passkey Domain after registering passkeys, remove all existing passkeys from Profile and re-register them.

Where are the log files?

Warden logs to stdout. In Docker, view them with docker logs -f <container_name>. Within the UI, go to Support → Server Log to see the last 500 log lines with auto-refresh.
